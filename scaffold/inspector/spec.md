# Inspector Spec — Art of Agents

---

# OVERVIEW — Purpose of the Inspector

The Inspector is a **workshop observability tool** designed to make the internal behavior of an AI agent visible and understandable.

It exists to support a core workshop principle:

> An agent is not a prompt.  
> An agent is a system with state, decisions, tool calls, and observable behavior.

The Inspector allows participants to:

- See the agent’s evolving **state**
- See every emitted **event**
- Observe tool calls
- Understand orchestration behavior
- Diagnose failures visually

This system is intentionally:

- In-memory only
- Non-persistent
- Non-production
- Architecturally simple

If the application restarts, all sessions are lost. That is acceptable for workshop use.

## 1.2 Sticky Global Header (Always Visible)

The header must appear in:

- Main Inspector  
- Chat pop-out  
- State pop-out  
- Events pop-out  

### Header Displays

Left side:

- **Agent name**
- **Session title**
- **Session ID (integer)**

Right side:

- SSE connection status  
- Current state revision (`rev`)  
- Event count  

The header guarantees that any view clearly indicates:

- Which agent is running  
- Which session is being observed

## 1.3 Session Navigator (Left Rail)

### Behavior

- “+ New Chat” button  
- Scrollable session list  
- Most recent sessions first  
- Clicking a session navigates to `/sessions/:id`  

### Collapsible

- Expanded ~280px  
- Collapsed ~56px  
- Toggle control  
- “+” remains visible when collapsed  

### Session IDs

- Integer  
- Monotonically incrementing  
- Globally assigned  
- Manually navigable via URL (e.g., `/sessions/7`)  

### Endpoints

```
POST /sessions
GET  /sessions
GET  /sessions/{id}/meta
DELETE /sessions/{id} (optional)
```

## 1.4 Chat Panel

### Behavior

- Displays full transcript  
- Input at bottom  
- Sends user message:

```
POST /sessions/{id}/messages
```

Transcript may be derived from event stream or explicit message objects.

---

## 1.5 State Viewer

### Behavior

- Displays latest state snapshot  
- Supports browsing previous revisions  
- Resizable  
- Scrollable  
- Pop-out capable  

### State Format

State is full markdown:

```md
# Agent State

- **ts:** 2026-02-14T18:22:31.123Z
- **rev:** 42
- **session:** 7

---
```

Only:

- `ts`  
- `rev`  
- `session` (integer ID)  

Everything after `---` is free-form markdown generated by the agent.

### Endpoints

Live stream:

```
GET /sessions/{id}/state/stream
```

Revision fetch:

```
GET /sessions/{id}/state/rev/{rev}
```

---

## 1.6 Events Viewer

### Behavior

- Append-only log  
- Entire history replayed on connect  
- Streams new events live  
- Expandable JSON view for `data`  
- Scrollable  
- Pop-out capable  

### Event JSON Format

```json
{
  "ts": "2026-02-14T18:22:31.123Z",
  "msg": "Tool call: GET /accounts/123",
  "data": { ... optional JSON ... }
}
```

### Endpoint

```
GET /sessions/{id}/events/stream
```

Behavior:

1. Replay entire event history  
2. Stream new events  

No resume cursors.  
No Last-Event-ID logic.

---

## 1.7 Pop-Out Routes

The following routes must exist:

- `/sessions/:id`
- `/sessions/:id/chat`
- `/sessions/:id/state`
- `/sessions/:id/events`

Each pop-out:

- Includes sticky header  
- Subscribes independently  
- Works without parent window

# PART 3 — BACKEND DESIGN

## 3.1 Core Principle

The backend revolves around a single aggregate:

> `InspectorSession`

An `InspectorSession` contains everything required to render the Inspector.

It is the only in-memory object needed for a session.

---

## 3.2 Session ID Strategy

```java
AtomicInteger sessionCounter = new AtomicInteger(0);
int id = sessionCounter.incrementAndGet();
```

Session IDs are integers.

---

## 3.3 Storage

```java
ConcurrentHashMap<Integer, InspectorSession>
```

All sessions live in memory.

Application restart clears all sessions.

---

## 3.4 InspectorSession Structure

`InspectorSession` is a mutable class containing:

Identity:

- `int sessionId`
- `String agentName`
- `String title`
- `Instant createdAt`
- `Instant lastUpdatedAt`

History:

- `List<InspectorMessage> messages`
- `List<InspectorEvent> events`
- `NavigableMap<Long, InspectorStateRevision> stateRevisions`

Counters:

- `AtomicLong nextEventSeq`
- `AtomicLong nextStateRev`

---

## 3.5 Records

Use Java records where possible:

```java
public record InspectorMessage(
    Instant ts,
    Role role,
    String text
) {}

public record InspectorEvent(
    Instant ts,
    long seq,
    String msg,
    JsonNode data
) {}

public record InspectorStateRevision(
    Instant ts,
    long rev,
    int sessionId,
    String markdown
) {}

public record InspectorSessionMeta(
    int sessionId,
    String title,
    String agentName,
    long stateRev,
    long eventCount,
    Instant lastUpdatedAt
) {}
```

---

## 3.6 InspectorSessionService

Responsibilities:

- Create session  
- List sessions  
- Retrieve session  
- Retrieve meta  
- Append message  
- Append event  
- Add state revision  
- Manage SSE streaming  

All session logic must be centralized here.

---

## 3.7 Controllers

Controllers must be thin wrappers over the service.

### SessionController

```
POST /sessions
GET  /sessions
GET  /sessions/{id}/meta
```

### ChatController

```
POST /sessions/{id}/messages
```

### EventsController

```
GET /sessions/{id}/events/stream
```

Replay entire event history, then tail.

### StateController

```
GET /sessions/{id}/state/stream
GET /sessions/{id}/state/rev/{rev}
```

---

## 3.8 Agent API Surface

The agent interacts only via:

```
appendEvent(sessionId, msg, JsonNode data)
setState(sessionId, markdown)
```

The agent is not responsible for storage or streaming.

# Final Mental Model

Agent:
- Emits events  
- Emits markdown state  

InspectorSession:
- Holds all session data  

InspectorSessionService:
- Manages sessions  
- Manages SSE  

Backend:
- In-memory only  
- Reset on restart  

Frontend:
- Subscribes  
- Renders  
- Shows agent + session context  
- Provides resizable layout and pop-out views
